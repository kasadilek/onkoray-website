# Accessibility

Baseline target: **WCAG 2.1 AA**. The skeleton ships the a11y primitives, reset rules, and lint/audit tooling; per-project design still has to meet the remaining criteria (colour contrast, focus visibility, motion sensitivity, etc.).

## Built-in primitives

### `<SkipLink>`

Added as the first body element by [`BaseLayout`](../src/layouts/BaseLayout.astro). Visually hidden until `:focus`. Default jumps to `#main-content`, label in German.

```astro
<SkipLink href="#main-content" label="Zum Hauptinhalt springen" />
```

### `<VisuallyHidden>`

Render content for screen readers only.

```astro
<button>
  Close
  <VisuallyHidden>dialog</VisuallyHidden>
</button>
```

### `<Icon>`

`astro-icon` wrapper with a11y baked in:

```astro
<!-- Decorative — aria-hidden -->
<Icon name="arrow-right" />

<!-- Meaningful — aria-label + role="img" -->
<Icon name="warning" label="Warnung" />
```

## Semantic HTML — the rules the agent audits against

These are the baseline expectations. The `web-design-guidelines` skill fetches a fuller set at audit time; this section is the shortlist that lives with the code.

### Native elements before ARIA

- Actions → `<button type="button">` (or `<button type="submit">` inside forms)
- Navigation → `<a href="…">`
- Never `<div onClick>` or `<span onClick>`. If you find yourself adding `role="button"` + `tabindex="0"` + keyboard handlers, switch to `<button>` instead.
- ARIA only when a native element doesn't exist for the pattern (e.g. disclosure, custom dropdown).

### Document structure

```html
<html lang="de-DE">
  <head>
    <!-- meta, title, links -->
  </head>
  <body>
    <!-- SkipLink -->
    <header>
      <nav aria-label="Primary"><!-- primary nav --></nav>
    </header>
    <main id="main-content" tabindex="-1">
      <article>
        <h1>Page title</h1>
        <section aria-labelledby="section-1-heading">
          <h2 id="section-1-heading">Section heading</h2>
          <!-- section content -->
        </section>
      </article>
    </main>
    <footer><!-- meta links, legal --></footer>
  </body>
</html>
```

- **One `<h1>` per page.** It's the page-level subject.
- **No level skipping.** h1 → h2 → h3. If the visual design calls for a small heading at a top level, size it with CSS, not with an incorrect tag.
- **Multiple `<nav>` or `<aside>`** → give each an `aria-label` so screen-reader users can tell them apart.
- **`<article>`** = self-contained piece you could syndicate (blog post, product card). **`<section>`** = themed group within a page.
- Don't nest landmarks of the same type unless semantically justified.

### Lists

- Use `<ul>` / `<ol>` for anything list-shaped: nav links, card grids, tag chips.
- When you remove list styling with `list-style: none`, Safari VoiceOver strips the list role. Re-add with `role="list"`. Our [`_reset.css`](../src/styles/_reset.css) already covers `ul[role="list"]` / `ol[role="list"]`.

### Interactive elements

- Every interactive element reachable via `Tab` in DOM order.
- `:focus-visible` must produce a visible ring (the default 3 px `currentColor` outline does this; don't override with `outline: none` alone).
- Buttons must have either visible text or `aria-label` + a decorative icon.
- Min 44×44 px touch target (already enforced by [`_reset.css`](../src/styles/_reset.css) on buttons and `[role="button"]`).

### Forms

```astro
<label for="email">E-Mail</label>
<input
  id="email"
  name="email"
  type="email"
  autocomplete="email"
  required
  aria-describedby="email-hint"
/>
<p id="email-hint">We'll never share this.</p>
```

- Label persistently visible. Placeholders disappear on input — never use as the sole label.
- `autocomplete` on name, email, phone, address fields.
- Group related fields with `<fieldset><legend>`.
- Errors announced: `aria-invalid="true"` + `aria-describedby="error-id"`.

### Images

- Meaningful content → `alt="Description of what's visible"`.
- Purely decorative → `alt=""` (empty, not absent).
- SVG icons → wrap with [`<Icon>`](../src/components/base/Icon.astro) — it emits `aria-hidden` (no label) or `aria-label` + `role="img"` (with label) automatically.
- Images of text → avoid; use real text styled with CSS.

### Tables

- `<th scope="col">` / `<th scope="row">` on headers.
- `<caption>` for a one-line summary.
- Never `<table>` for layout — use CSS Grid / Flexbox.

### Content patterns

- Dates in `<time datetime="2026-04-22">` so assistive tech and parsers get a machine-readable value.
- Addresses in `<address>`.
- Code in `<code>` / `<pre><code>`; key commands in `<kbd>`.
- Language shifts in content: `<span lang="en">`.

## Defaults the skeleton enforces

- `<main id="main-content" tabindex="-1">` — skip-link target
- `button`, `[role="button"]`, `input[type="submit"]`, `summary` — `min-height: 44px` (WCAG 2.5.5)
- `:focus-visible` — `outline: 3px solid currentColor; outline-offset: 2px`
- Reduced-motion — global safety-net in [`_reset.css`](../src/styles/_reset.css) collapses all animations to 0.01s under `prefers-reduced-motion: reduce`
- `ul[role="list"]`, `ol[role="list"]` — Safari VoiceOver keeps list semantics even with `list-style: none`

## Common pitfalls

Tooling- and rendering-level traps that are easy to miss because the visual result looks fine but axe/pa11y will fail CI. Each one has been encountered in a real project built on this skeleton; worth skimming before writing custom widgets or styling new components.

### Opacity-muted text

`opacity: 0.5` on text almost always fails axe's `color-contrast` rule. axe blends the foreground with the rendered background and checks effective luminance — at `opacity: 0.5` the result typically lands between 3.0:1 and 3.5:1, below the AA threshold of 4.5:1 for body text.

Use a solid muted colour instead. Example: `color: #555250` on `#F5F2EC` gives a 6.94:1 ratio (AA comfortably, almost AAA). Pick the hex once and reuse it via a CSS custom property.

```css
/* ❌ Don't */
.label {
  color: var(--fg);
  opacity: 0.5;
}

/* ✅ Do */
.label {
  color: var(--fg-muted); /* precomputed for ≥4.5:1 on --bg */
}
```

### Scoped-style specificity trap

This project sets [`scopedStyleStrategy: "where"`](../astro.config.mjs) on Astro so scoped selectors don't inflate specificity. A side effect: inside the same scoped block, an element-selector rule (`.parent a`, specificity 0,1,1) beats a class-only BEM child (`.parent__link`, specificity 0,1,0) — even when the class is the more specific hand-written one.

Real example: `.site-nav__logo { color: cream }` was overridden by a sibling `.site-nav a { color: inherit }` rule in the same file, so the logo rendered in the body text colour (black on black).

Rule of thumb: inside scoped styles, don't write element selectors for tags that also appear as BEM children. Style the specific class directly, or drop the inherited-colour reset.

### Icon-only buttons (non-BMP glyphs)

Buttons whose only text content is a single glyph like `→` or `✕` trigger axe's `color-contrast` rule with messageKey `nonBmp` — "Element content contains only non-text characters". The rule can't determine whether the glyph is even meant to be read.

Fix: move the glyph to CSS pseudo-content and put the accessible label inside [`<VisuallyHidden>`](../src/components/a11y/VisuallyHidden.astro).

```astro
<button class="next-btn" type="button">
  <VisuallyHidden>Nächster Slide</VisuallyHidden>
</button>
```

```css
.next-btn::before {
  content: "→";
}
```

### Disclosure / accordion pattern

Don't reach for `<div role="listitem" tabindex="0" aria-expanded="true">` — it trips two axe rules at once (`no-noninteractive-tabindex`, `role-supports-aria-props`). Use the native interactive element for the header:

```astro
<ul role="list">
  <li>
    <button type="button" aria-expanded="false" aria-controls="panel-1">
      Header text
    </button>
    <div id="panel-1" hidden>Body content</div>
  </li>
</ul>
```

`tabindex="0"` belongs on natively focusable elements only. `aria-expanded` belongs on `<button>` or `<summary>` — not on list items.

### Layered decorative effects (`axe bgOverlap`)

When a design stacks absolutely-positioned elements — split-colour heroes with `clip-path` reveals, duplicated headlines with `mix-blend-mode`, decorative SVG pattern overlays — axe can flag `color-contrast` with messageKey `bgOverlap` even if the visible pixels pass contrast. Axe walks the DOM to find the background, not the rendered canvas, so it can't resolve which layer is effectively visible.

First try: mark the decorative duplicates with `aria-hidden="true"` and a single accessible copy of the text (the [`<VisuallyHidden>`](../src/components/a11y/VisuallyHidden.astro) pattern works for this).

As a last resort, escalate by hiding them from the axe run via pa11y-ci's `hideElements` option — only for elements that are provably decorative; never to silence a real failure:

```json
{
  "defaults": {
    "hideElements": ".decorative-layer"
  }
}
```

## Customising the focus ring

Override CSS custom properties (usually in `src/styles/_root.css` or a theme file):

```css
:root {
  --focus-ring-color: #ff5500;
  --focus-ring-width: 4px;
  --focus-ring-offset: 3px;
}
```

ESLint runs `eslint-plugin-jsx-a11y` rules on `.astro` files — redundant roles, missing alt text, inaccessible form labels caught before commit. See [`eslint.config.mjs`](../eslint.config.mjs).

## Runtime patterns

### Keyboard navigation

- Every interactive element is reachable via Tab
- Modal/drawer implementations must trap focus and return it on close
- `Escape` closes dismissible UI

### Form labels

```astro
<!-- Never -->
<input type="email" placeholder="Email" />

<!-- Always -->
<label for="email">Email</label>
<input id="email" type="email" required />
```

### Colour contrast

Pre-launch: run every foreground/background combo through [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/). Min: 4.5:1 for body text, 3:1 for large text or UI components.
